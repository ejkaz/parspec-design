# Parspec Slides

Forked from `zarazhangrui/frontend-slides` and rewired for Parspec brand. Reads `../parspec-design/skills/parspec-design/design-model.yaml` for tokens; emits zero-dependency single-HTML decks with full Parspec primary axis applied; preserves all viewport invariants and the PPT / PDF / Vercel pipeline.

**Two generation modes** (chosen at Phase 2.5):
- **Sequential** — single Claude agent writes the whole deck slide-by-slide. Default. Best for short decks (≤ 8 slides), drafts, exploratory work, or low context-budget situations.
- **Parallel** — conductor agent plans, fans out to N writer subagents (one per slide), reviewer agent does editorial pass, targeted revisors fix issues, conductor reassembles. Best for ≥ 8-slide decks where wall-time matters or where editorial cohesion across many slides is harder to hold in a single agent's context.

## Identity

Parspec sells AI-native quoting / submittals / project-management software to **Construction Materials Distributors** (Sonepar, Wesco, Border States, Rexel, Graybar). Decks generated by this skill are for board updates, sales, customer onboarding, internal townhalls, investor materials. Default surface mode is `brand_artifacts` (dark, paper-warm panels, Brand Orange CTAs) per `design-model.yaml v2.0.0`.

## Locked invariants (non-negotiable)

From `design-model.yaml.invariants:` — `parspec-craft` enforces them on every output regardless of mode.

1. **Viewport fitting** — every `.slide` has `height: 100vh; height: 100dvh; overflow: hidden;`. No scrolling within slides.
2. **Clamp typography** — ALL font sizes and spacing use `clamp(min, preferred, max)`.
3. **viewport-base.css inlined** — full file contents inlined in `<style>` of every output.
4. **Reduced motion** — every output includes `@media (prefers-reduced-motion: reduce)` override.
5. **CTA monopoly** — Brand Orange `#FFA72B` is the only primary CTA color. Never decorative.
6. **Red markup monopoly** — Engineering Red `#B91C1C` only in error/markup/REV-stamp roles.
7. **Warn ≠ brand** — `roles.warn` resolves to Caution Amber `#B45309`, never `brand.500`.
8. **Orange-type-on-light forbidden** — Brand Orange as type fails AA on Off-white. Use `brand.800 #8C5800` for accent type on light.
9. **Paper-warm panel only** — surface_paper `#FAF7F0` never used as a slide background.
10. **Steel is doc register** — `steel.500 #205991` only in info / link / doc-metadata roles.
11. **Density limits** per slide type (see table below).

### Content density per slide

| Slide type | Maximum |
|---|---|
| Title | 1 heading + 1 subtitle + optional eyebrow tagline |
| Content | 1 heading + 4–6 bullets OR 1 heading + 2 paragraphs |
| Feature grid | 1 heading + 6 cards max (2×3 or 3×2) |
| Code | 1 heading + 8–10 lines of code |
| Quote | 1 quote (max 3 lines) + attribution |
| Image | 1 heading + 1 image (max 60vh height) |
| Metric | 1 heading + up to 3 large stats |

## Phase 0 — Detect mode

Determine what the user wants:

- **Mode A: New Parspec deck** (default) — Phase 1
- **Mode B: PPT conversion** — convert a `.pptx` file. Phase 4 first, then back to Phase 3
- **Mode C: Enhance** — improve an existing HTML deck per density limits
- **Mode D: Non-brand exploratory** — `--preset <name>` flag uses `references/legacy-presets.md` instead of design-model.yaml

## Phase 1 — Brand load + content discovery

1. Load brand tokens from `../parspec-design/skills/parspec-design/design-model.yaml`. Extract:
   - `primary_axis_preserved.*` — required CSS variables
   - `tokens.colors.dark` and `tokens.colors.light`
   - `tokens.typography.*`
   - `roles.*` — semantic role bindings (cta, accent_type_on_*, info_*, etc.)
   - `surfaces.*` — mode bindings
   - `voice.do` and `voice.avoid`
2. Confirm scope with one `AskUserQuestion`: purpose, audience, length, content readiness.
3. If user provided images, evaluate them.

## Phase 2 — Style is locked

Default Mode A: skip style discovery. Parspec brand from design-model.yaml is the style. Confirm direction with user in one short summary.

Mode D: read `references/legacy-presets.md`, follow original frontend-slides flow.

## Phase 2.5 — Generation mode + speed/quality toggles ★ NEW

If deck length (from Phase 1) is **≤ 8 slides**: default to **sequential** mode (skip the parallel-mode toggles and proceed to Phase 3-sequential).

If deck length is **≥ 8 slides** OR user explicitly requests parallel: ask the user **two upfront questions** via `AskUserQuestion`:

### Q1 — Reviewer model level

| Choice | Speed | Quality | When to pick |
|---|---|---|---|
| `skip` | fastest (no Round 2/3) | none | low-stakes drafts, internal scratch decks |
| `haiku` | fast, cheap | catches obvious issues | short decks, rapid iteration |
| `sonnet` (recommended default) | balanced | catches narrative + density issues | most decks |
| `opus` | slowest | catches subtle voice + cohesion issues | board decks, customer-facing, IPO roadshow tier |

### Q2 — Visual review

| Choice | Speed | Quality | When to pick |
|---|---|---|---|
| `off` (default) | none | manifest-only | most decks |
| `on` | +30–60s render + vision tokens | catches contrast failures, layout regressions, anti-AI-slop visual | board decks, social-shareable, high-fidelity required |

Record the answers in `plan.yaml.toggles` for Round 0 conductor consumption.

## Phase 3 — Generate

### Phase 3-sequential (default for ≤ 8 slides; Mode D always uses this)

Single agent reads in order:
1. `viewport-base.css` — inline full contents
2. `references/layouts.md` — copy paste-ready layout(s) for requested slide types
3. `assets/template.html` — Parspec brand seed
4. `animation-patterns.md`
5. `html-template.md`

Generate a single self-contained `.html` file with:
- `<head>`: Montserrat from Google Fonts
- `<style>`: full viewport-base.css + `:root` populated from design-model.yaml + slide CSS
- `<body>`: sequence of `<section class="slide">` blocks
- `<script>`: nav controller (or omit if user opted out)

### Phase 3-parallel (≥ 8 slides; user-selected)

**Round 0 — Plan (sequential, conductor)**

Use the **Agent** tool with `subagent_type: general-purpose`, model from Phase 2.5 Q1 (use `sonnet` if conductor model not user-specified). Prompt: `references/conductor-prompt.md`. Output: `plan.yaml` + `prelude.css` (read-only inputs for Round 1).

**Round 1 — Parallel writers (parallel × N)**

Fire N subagents in parallel (single message, multiple Agent tool calls). For each slide N, the writer subagent's prompt is `references/subagent-writer-prompt.md` with the slide's `plan.yaml` entry + adjacent slide briefs + `prelude.css`. Each writes:
- `slides/slide-NN.html` — `<section class="slide">…</section>` block
- `slides/slide-NN.manifest.yaml` — sidecar metadata per `manifest.schema.yaml`

Wait for all N to complete before Round 2.

**Round 2 — Editorial review (sequential reviewer)**

If Q1 answer was `skip`: skip Round 2 + 3 entirely; go to Round 4.

Otherwise: spawn one reviewer subagent with the model selected at Q1. Prompt: `references/reviewer-prompt.md`. Reviewer reads all manifests + scans HTML lightly + outputs `review/r1.yaml` with PASS/REVISE/BLOCK per slide + cross_slide_notes.

If `decision.ready_to_assemble: true` → Round 4.
If `decision.revisions_needed: [...]` → Round 3.
If `decision.blocks: [...]` → surface to user.

**Round 3 — Targeted revisions (parallel × revise-list)**

For each slide in `decision.revisions_needed`, fire a revising subagent (same identity / model class as Round 1). Prompt: `references/revisor-prompt.md`. Each writes:
- `slides/slide-NN.v2.html`
- `slides/slide-NN.v2.manifest.yaml`

**Round 4 — Reassembly (sequential, conductor)**

Conductor stitches:
- Pick `slide-NN.v2.html` if exists, else `slide-NN.html`
- Inject prelude (viewport-base.css + :root + chrome CSS)
- Inject nav controller JS
- Write `deck.html`
- Run `parspec-craft` lint as gate (errors → conductor reports to user)

**Round 5 — Visual review (optional, sequential, vision-capable model)**

Only if Q2 answer was `on`. Render `deck.html` to per-slide PNGs via Playwright (reuse `scripts/export-pdf.sh` machinery). Spawn vision-capable reviewer (must be `opus` or `sonnet` with image input). Prompt: `references/visual-review.md`. Reviewer outputs `review/visual-r1.yaml`.

If visual-pass → ship. If visual-revise → Round 3 again on flagged slides. If still failing after one re-revise → surface to user.

**Convergence**: cap total rounds at 5 (Round 1 → Review → Revise → Reassemble → Visual → maybe Re-Revise). If not converged, surface to user with current state.

## Phase 4 — PPT conversion

(unchanged from sequential mode)

1. `python scripts/extract-pptx.py <input.pptx>` → JSON + assets
2. Show user extracted summary
3. Confirm/edit content
4. Proceed to Phase 3 (sequential or parallel based on slide count)

## Phase 5 — Delivery

Open generated HTML in browser. Summarize file path, slide count, navigation, customization entry points. If parallel mode: also surface review-round count and any concerns flagged by reviewers.

## Phase 6A — Deploy to Vercel (optional)

`bash scripts/deploy.sh <html-file>` — unchanged.

## Phase 6B — Export to PDF (optional)

`bash scripts/export-pdf.sh <html-file> [output.pdf] [--compact]` — unchanged.

## Phase 7 — Lint via parspec-craft

After generation (any mode), suggest:
```
parspec-craft lint <generated.html>
```
Catches v1 secondary leakage, missing locked-axis tokens, density overflows, non-Montserrat fonts before delivery.

## Voice (from design-model.yaml `voice.*`)

**Do**: lead with metrics, ALL CAPS section headers, name customers (Sonepar/Graybar/Rexel/Wesco/Border States), MEP industry vocabulary (quoting, submittals, O&M, ERP), terse.

**Avoid**: consumer-SaaS hype, generic startup verbs, wellness adjectives, lorem ipsum, emoji.

## What this skill does NOT do

- Maintain the brand — that's `parspec-design`
- Lint final output — that's `parspec-craft` (run after generation)
- Generate non-slide surfaces (memos, landing pages, dashboards) — future skills, not v1
